# 변명아카이브 기능 제안서

> "비난 대신 관찰, 후회를 성장으로" — 이 컨셉을 유지하면서 사용자가 기록을 **습관화**하고, 기록에서 **실제 인사이트**를 얻도록 설계하는 방향으로 제안합니다.

---

## 현재 서비스 구조 요약

| 영역 | 구현 상태 |
|------|-----------|
| 익명 인증 | ✅ |
| 기록 CRUD | ✅ |
| 실수 유형 / 감정 / 강도 태깅 | ✅ |
| nextActionDone 완료 토글 | ✅ |
| 7일 / 30일 패턴 리포트 (Top 유형/감정/트리거) | ✅ |

현재는 **기록하고 패턴 보는 것**까지만 가능한 구조예요.
아래 제안은 "기록 → 인사이트 → 행동 변화"의 흐름을 완성하는 방향입니다.

---

## 제안 기능

### 1. 미완료 다음 행동 모아보기

**왜 필요한가**
`nextAction`을 작성해도 `nextActionDone`을 다시 켜러 상세 페이지로 들어가야 해요. 실제로 "내가 약속한 것들을 지키고 있나?" 를 한눈에 볼 수 없어요.

**설계**
- 기록 목록 화면에 **"미완료" 필터 탭** 추가
- 홈 화면에 "아직 못 한 것들 N개" 배너 표시
- 기존 `GET /records` API + 클라이언트 필터링으로 구현 가능 (백엔드 변경 없음)

```
[홈 화면]
┌─────────────────────────────┐
│ 아직 못 한 것들 3개 있어요   │
│ 지금 확인하기 →              │
└─────────────────────────────┘
```

**구현 난이도**: 낮음 (프론트엔드만)

---

### 2. 연속 기록 스트릭

**왜 필요한가**
기록 자체가 목적이 아니라 **습관**이 목적이에요. 며칠 연속으로 기록했는지 보여주면 이탈을 줄이는 데 효과적이에요. 운동 앱이나 언어 학습 앱에서 검증된 방법이기도 해요.

**설계**
- 홈 화면 상단에 "🔥 5일 연속 기록 중" 표시
- 클라이언트에서 기록 날짜 목록을 분석해 연속 일수 계산
- 기록이 없는 날은 스트릭 끊김

```
[홈 화면 헤더]
🔥 5일 연속 기록 중
이번 주: ● ● ● ● ● ○ ○  (오늘 포함 5일)
```

**구현 난이도**: 낮음 (프론트엔드 + `GET /records` 전체 날짜 목록 활용)

---

### 3. 감정 강도 트렌드 그래프

**왜 필요한가**
현재 패턴 리포트는 "어떤 유형이 많냐"만 알 수 있어요. 하지만 **감정 강도(intensityLevel)가 시간에 따라 어떻게 변하는지**가 사용자에게 훨씬 실질적인 인사이트를 줘요. "이번 주는 저번 주보다 강도가 낮아졌어요"가 성장의 증거가 됩니다.

**설계**
- 패턴 리포트 화면에 "강도 변화" 섹션 추가
- 날짜별 intensityLevel 평균을 라인/바 차트로 표시
- 백엔드 변경 없이 클라이언트에서 `GET /records` 데이터로 직접 계산

```
[패턴 리포트]
평균 감정 강도 추이
5 │
4 │   ●
3 │ ●   ●   ●
2 │         ●
  └─────────────
  월  화  수  목  금
```

**구현 난이도**: 중간 (`react-native-chart-kit` 또는 `victory-native` 라이브러리 추가 필요)

---

### 4. 주간 요약 카드

**왜 필요한가**
매주 초에 "지난 주 나는 어땠나"를 한 장의 카드로 보여주면 리텐션과 자기 인식 모두 올라가요. 저널 앱에서 많이 쓰는 패턴이에요.

**설계**
- 홈 화면 또는 리포트 화면에 "지난 주 요약" 카드 표시
- 항목: 총 기록 수 / 가장 많은 실수 유형 / 완료한 nextAction 비율
- 클라이언트에서 직접 계산 가능

```
[지난 주 요약 카드]
─────────────────────
지난 주 (3/24 ~ 3/30)

기록 7개  |  완료율 57%
가장 많이: 미루기 (4회)
감정: 불안 > 죄책감
─────────────────────
```

**구현 난이도**: 낮음 (프론트엔드만)

---

### 5. 기록 검색

**왜 필요한가**
기록이 쌓이면 "그때 그 상황" 을 다시 찾고 싶은 니즈가 생겨요. 현재는 페이지네이션만 있고 검색이 없어서 오래된 기록을 찾을 방법이 없어요.

**설계**
- 기록 목록 상단에 검색창 추가
- 검색 대상: situation, cause, myAction
- **백엔드에 `GET /records?q=키워드` 파라미터 추가 필요**

```
[기록 목록]
┌──────────────────────┐
│ 🔍 기록 검색...       │
└──────────────────────┘
```

**구현 난이도**: 중간 (백엔드 API 변경 필요)

---

### 6. 푸시 알림 (기록 리마인더)

**왜 필요한가**
기록 앱의 가장 큰 적은 **망각**이에요. 매일 저녁 "오늘 기록 안 했어요" 알림 하나로 DAU를 크게 올릴 수 있어요.

**설계**
- 설정 화면에서 알림 시간 설정 (기본: 오후 10시)
- 그날 기록이 없으면 알림 발송
- `expo-notifications` 로컬 알림으로 구현 (서버 불필요)

**구현 난이도**: 중간 (`expo-notifications` 추가 필요)

## 우선순위 제안

| 순위 | 기능 | 이유 |
|------|------|------|
| 1 | 미완료 nextAction 모아보기 | 기존 API 그대로, 핵심 가치와 직결 |
| 2 | 연속 기록 스트릭 | 습관 형성 → 이탈 방지 |
| 3 | 주간 요약 카드 | 성취감 제공, 구현 빠름 |
| 4 | 감정 강도 트렌드 | 차별화 포인트, 라이브러리 필요 |
| 5 | 기록 검색 | 데이터 쌓인 후 가치↑, 백엔드 필요 |
| 6 | 푸시 알림 | 효과 크지만 권한 처리 복잡 |

---

## 제외 검토 기능

| 기능 | 제외 이유 |
|------|-----------|
| 소셜 공유 | 서비스 컨셉(반성 기록)과 공유는 어울리지 않음 |
| 랭킹/리더보드 | 비교가 아닌 자기 성찰이 핵심 |
| AI 피드백 | 현재 단계에서 오버엔지니어링, 추후 고려 |

---

## 구현 시 공통 고려사항

1. **오프라인 대응** — 기록은 로컬에 임시 저장 후 온라인 시 동기화 (현재 없음)
2. **날짜 표시 로컬라이즈** — occurredAt은 항상 UTC로 저장되는데 UI에서 로컬 시간으로 변환 필요
3. **빈 상태(Empty State) 개선** — 각 화면의 데이터 없음 상태를 더 친근하게 (현재 텍스트만 있음)
